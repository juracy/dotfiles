# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Juracy Filho's personal dotfiles, managed with GNU Stow. There is no build/compile step — the repo *is* the
configuration; changes take effect once (re-)stowed into `$HOME`, or immediately for tools that hot-reload
their config (Hyprland, zsh, kitty, etc).

## Repository layout — Stow packages

Each top-level `*.package` directory is a Stow package: its internal structure mirrors `$HOME` exactly, so
`config.package/.config/hypr/...` stows to `~/.config/hypr/...`.

- `config.package/` — `~/.config/*` for most apps (Hyprland, DankMaterialShell, kitty, zed, rofi, yazi, tmux, etc.)
- `home.package/` — dotfiles that live directly in `$HOME` (`.gitconfig`, `.gnupg`, etc.)
- `zsh.package/` — `.zshrc`, `.zshenv`, and `.local/bin/zsh/*.zsh` / `.local/share/zsh` helper scripts sourced from `.zshrc`
- `local.package/` — `~/.local/*` (non-zsh)
- `assets.package/` — `~/.assets/*`

Stow each package from the repo root, e.g.:
```
stow -t "$HOME" config.package home.package zsh.package local.package assets.package
```

### Ignore rules and personal/machine-specific overrides

- `.stow-local-ignore` (root) and `home.package/.stow-local-ignore` list paths Stow must skip — mostly
  machine-generated or host-specific files (colors, monitor layouts, workspaces, theme files, DMS backups).
- Personal/secret values are split into separate `*-personal.*` / `*-personal.conf` files that are
  git-ignored (see root `.gitignore`) but read by the corresponding non-personal config. When adding a new
  personal override, follow the existing naming convention (`NN-name-personal.lua`/`.conf`) and add a
  `.sample` file alongside it (see `hypr.d/15-env-personal.lua.sample`) so the structure is discoverable
  without leaking machine-specific data.
- Git identity (`user.name`/`user.email`) is intentionally kept out of `home.package/.gitconfig` and must be
  set in `$HOME/.gitconfig-personal` instead (referenced via `[include]` in `.gitconfig`).

## Hyprland configuration (`config.package/.config/hypr/`)

This is the most actively developed part of the repo and uses Hyprland's Lua config support (Hyprland
0.54+/0.55+, via DankMaterialShell/"dank linux"):

- `hyprland.lua` is the entry point; numbered files under `hypr.d/` (`00-apps.lua`, `10-env.lua`,
  `20-exec.lua`, `6X-input.lua`, `7X-keybindings-*.lua`, `8X-rules.lua`, ...) are loaded in order — the
  numeric prefix controls load order, so new snippets should be inserted with a prefix matching their
  category (env vars in the 10s, exec in the 20s, keybindings in the 70s, window rules in the 80s).
- `hypr.d/*-personal.lua` files hold host-specific env/keybindings/rules and are git-ignored; edit the
  matching `.sample` file when changing the shape of a personal override.
- `dms/` holds DankMaterialShell-generated/managed Lua (binds, colors, cursor, layout, outputs, window
  rules) — `dms/colors.lua` and `dms/outputs.lua` are regenerated by DMS and git-ignored; don't hand-edit
  generated values there.
- `utils/` contains standalone Lua helper scripts (e.g. `swap-workspaces.lua`, `zoom.lua`) invoked from
  keybindings.
- `.luarc.json` at the repo root declares the `hl` global so `lua-language-server` doesn't flag Hyprland's
  injected API in these Lua files.

## Spelling/lint config for editors

`cspell.json` (root) and `.vscode/settings.json` hold project-specific spellcheck word lists — extend
these rather than adding inline spellcheck-disable comments when a legitimately new term (tool name, CLI
flag) trips the checker.
